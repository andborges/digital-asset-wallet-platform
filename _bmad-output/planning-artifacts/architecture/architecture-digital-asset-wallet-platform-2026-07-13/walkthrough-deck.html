Digital Asset Wallet Platform — Architecture Walkthrough

The ledger side, done rigorously.

Deposits, withdrawals, and balances for ETH & USDC on Base and Arbitrum — the orchestration layer every custody build ends up hand-rolling: confirmation tiers, reorgs, nonces, idempotency, reconciliation.

No vendor covers it. The bet is that rigor is the product: tests that inject reorgs, a reviewed threat model, and a ledger that is always explainable against the chain.

INVARIANT

A credited balance is never reversed by a chain event — and no money moves twice.

Why this is hard

Four problems, each with a real-cost failure mode

01 · TIERS

Deposits are not binary

An L2 transaction passes sequencer receipt → safe → finalized (~13–20 min). Credit early and a reorg becomes a double-credit.

02 · RETRIES

Withdrawals must never happen twice

Crashes, retries, and duplicate calls are normal operation. Without idempotency and explicit state, they become duplicate payouts.

03 · FEES

Fees are chain-specific

L2 execution fee + amortized L1 data fee, surfaced differently on Arbitrum and Base. Naive L1-style estimation systematically undercharges.

04 · DRIFT

The ledger drifts from the chain

Missed deposits, stuck withdrawals, double-counts. Without independent reconciliation, customers find drift before operators do.

Design paradigm

Hexagonal core · one binary · five roles

The domain core — ledger, state machines, policy — imports nothing. Adapters implement its ports. Chain isolation is structural, not conventional: nothing outside the EVM adapter may import go-ethereum or name a chain ID, enforced in CI (AD-1).

One compiled binary, walletd, run as separate OS processes per role — api · watcher · broadcaster · recon · dispatcher — that share exactly one thing: Postgres. No RPC between processes, no broker, no shared memory (AD-2).

The money model

Double-entry, derived balances

Every movement is a balanced journal entry across a fixed account taxonomy. Balances are derived — always recomputable from postings. One cause per entry, enforced by a unique constraint (AD-3).

// deposit credited at finality — one atomic transaction (AD-4)

CR customer:9f31…:available:usdc 250.000000
cause = chain_event(base, 0x8c4a…, log 3) + outbox: deposit.credited
// sweep — treasury logistics, customers never touched (AD-9)

Pre-finality records are visible as pending and safely reversed on reorgs. The watcher is the sole writer of deposit rows — including the credit itself (AD-6). Crediting policy is a per-(chain, asset) table; v1 pins every entry to finalized (AD-7).

Recovery after downtime is a cursor rescan; over-rescanning is harmless because chain events dedupe on (chain, tx_hash, log_index) (AD-5).

Address generation

One address per customer, every EVM chain, zero keys

Deposit addresses are counterfactual CREATE2 forwarders: computed off-chain from (factory, salt, init code), deployed lazily at sweep time. The same factory sits at the same address on every chain via the canonical deterministic deployer — so the address holds on Base, Arbitrum, and any future EVM chain (AD-8).

No per-customer key material

The only signing keys in the system are the hot wallet's. No HD master seed, no sweep-key fleet.

Immutable by design

Factory address, init code, and salt scheme never change once live — changing any of them changes every customer address.

Persistent contracts

EIP-6780 ended destruct-and-redeploy; forwarders are persistent, modeled on BitGo's ForwarderV4 pattern.

Policy checks run before signing: balance covers amount + fee, destination well-formed, threshold routing applied (FR18). Every transition is a persisted, guarded step — crash mid-machine and the process resumes from the row (AD-6).

Exactly one broadcaster per chain — enforced by a Postgres advisory lock, not deployment discipline — signs and sends everything outbound, withdrawals and sweeps alike, allocating nonces from persisted state (AD-11). Consumers never see nonces, gas, or raw transactions.

Correctness machinery

Exactly-once, enforced by the database

AD-4One transaction per observable change. Money changes carry postings + transition + outbox event atomically; state-only changes carry transition + event. Nothing observable happens outside such a commit.
AD-5Idempotency by unique constraint, not by care. API keys table with stored responses; chain events on (chain, tx_hash, log_index); one journal entry per cause. Retries hit constraints, never double-applies.
AD-13Webhooks ride the outbox. At-least-once, HMAC-signed, event ID = outbox row minted in the same transaction as the change it announces. Consumers dedupe on it.
AD-11Single-writer broadcasting. Nonce races removed structurally — one process per chain owns outbound, with an advisory lock making "one" true at runtime.

NFR1 and NFR2 aren't aspirations reviewed in code review — they fall out of schema constraints and transaction boundaries.

Failure & degraded modes

Outages are a mode, not an incident

REORGS

History changes safely

Pre-finality records reverse or re-credit; credited balances are untouchable by construction. Sequencer reordering and L1 batch reorgs are detected and absorbed (FR12–13).

SEQUENCER HALT

Explicit chain status (AD-15)

Liveness derived from watcher heartbeat and cursor staleness, exposed in the API. Deposits stay pending; withdrawals are accepted and queue — never rejected for liveness. Base halted twice in 2026; this is normal weather.

CRASH

Resume from rows alone

Every process restarts from persisted state: cursors rescan, machines resume, holds stand. No operator reconstruction, ever (NFR8).

DRIFT

Independent reconciliation (AD-12)

Streaming break-detection + batch deep pass on its own RPC provider. Read-only observer: it alarms, it never fixes. Seeded-fault tests prove the alarm actually fires.

Security posture

The key never exists in software

AD-10One signing boundary. All signing goes through the core's Signer port. Production: AWS KMS secp256k1 — the key is generated in, and never leaves, the HSM-backed service. Dev/test: software signer behind the same port.
NFR16The Bybit lesson. $1.5B moved through a compromised third-party signing UI. Here, raw transactions are constructed and verified by the platform itself — no third-party UI in the path.
NFR12The threat model ratifies, or the decision changes. The written threat model is a deliverable with the same status as code — and AD-10 binds itself to it: a risk KMS can't mitigate revises the AD, not the workaround.

Honest edge, documented: funds sent to a customer address on an unsupported chain are recoverable only where the deterministic factory can be redeployed — stated plainly in consumer docs rather than discovered in an incident.
